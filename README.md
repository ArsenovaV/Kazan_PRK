# Казань — Природно-рекреационный каркас

Интерактивная аналитическая веб-карта доступности природно-рекреационного каркаса (ПРК) Казани. Показывает зелёные зоны города по категориям, изохроны и буферы пешеходной доступности, обеспеченность жилых территорий зеленью, а также позволяет смоделировать размещение нового объекта ПРК и мгновенно оценить охватываемое им население.

## Стек

- **MapLibre GL JS** — рендеринг карты и работа с векторными слоями (GeoJSON)
- **Nominatim (OpenStreetMap)** — геокодер для поисковой строки
- **HTML + CSS + JavaScript** — без сборщиков и фреймворков
- **QGIS** — пространственный анализ (зоны и сетевая доступность, население)
- **Python (geopandas)** — агрегированная статистика по объектам
- Базовая подложка — `positron-nolabels` (CartoDB)
- Шрифт — Inter

## Данные и источники

| Слой / показатель | Источник | Поля |
|---|---|---|
| Полигоны зелёных зон ПРК | OpenStreetMap + ручная доразметка по категориям МНГП | `00_Назначение (категория) объекта`, `00_Название объекта` |
| Буферы пешеходной доступности | Рассчитаны в QGIS по нормативам МНГП (по категории объекта) | — |
| Изохроны пешеходной доступности | Сетевой анализ в QGIS на графе дорожной сети OSM | — |
| Точки многоквартирных домов с населением | ГИС ЖКХ (площадь жилого фонда) + средняя обеспеченность жильём в Казани за 2025 г. | `POPUL` |

## Методология

**Зоны и сетевая доступность — QGIS.** Изохроны построены по графу дорожной сети OSM с пороговыми значениями пешеходной доступности из МНГП, индивидуально для каждой категории объектов ПРК (парк, сад, сквер, бульвар и т. д.). Буферы — упрощённая радиальная альтернатива изохронам.

**Численность населения — QGIS.** Так как открытых данных о точной численности жителей по адресам нет, для каждой точки МКД из ГИС ЖКХ численность населения оценена через среднюю обеспеченность жильём в Казани (2025) и суммарную жилую площадь дома: `POPUL ≈ S_жил / S_на_чел_среднее`. Полученные точки используются в инструменте «Добавление нового объекта» — суммируются те, что попали в буфер.

**Обеспеченность жилых территорий — QGIS.** Для каждого жилого полигона (`landuse=residential`) рассчитано отношение суммарной площади объектов ПРК в зоне пешеходной доступности к численности населения в полигоне. Полученное значение классифицировано на три градации: высокая / средняя / низкая (или не обеспечены).

**Агрегированная статистика — Python.** Суммарная площадь объектов ПРК, разбивка по типам объектов и общие показатели для блока статистики в правой панели посчитаны скриптом на Python (geopandas) по итоговым слоям.

**Веб-визуализация — MapLibre GL JS.** Готовые слои выгружены в GeoJSON и подключены к карте. Логика переключений, фильтрация по категориям зелёных зон, кастомная масштабная линейка с русскими подписями, инструмент моделирования нового объекта (буфер + подсчёт населения «на лету») реализованы на ванильном JavaScript.

## Интерфейс

**Шапка:** название проекта на татарском и русском, поисковая строка с геокодером Nominatim (подсказки от 3 символов, найденная точка отмечается пином).

**Левая панель — легенда и переключение слоёв:**
- Зелёные зоны с раскрывающимся списком из 15 категорий (ООПТ, бульвары, скверы, парки, набережные и др.); включение/выключение всех разом или по одной
- Буферы доступности (по нормативам МНГП по категориям)
- Изохроны доступности (по нормативам МНГП по категориям)
- Обеспеченность жилых территорий (три градации)
- Приоритетные территории для увеличения доступности
- Численность населения с цветовой шкалой
- Картодиаграммы — отношение видов объектов, обслуживающих зелёные зоны
- Границы города и исследуемой территории (всегда отображаются)

**Карта:** зум-контролы, кастомная масштабная линейка в метрах/километрах, popup по клику на полигон.

**Правая панель:**
- Блок статистики — крупная карточка с долей населения в зоне доступности + сетка 2×2 (общее население, население в зоне доступности, число объектов, суммарная площадь)
- Инструмент **«Добавление нового объекта»**: выбор типа (парк 1200 м / сад 600 м / сквер 400 м / бульвар 400 м), клик по карте → пин + буфер пешеходной доступности → суммарное население внутри буфера крупной цифрой

## Запуск

Файлы статические, никакой сборки не требуется. Любой локальный HTTP-сервер подойдёт — открыть `index.html` напрямую через `file://` нельзя, потому что `fetch` для GeoJSON будет заблокирован браузером.

## Что получилось и чем гордимся

- **Связали разрозненные источники** (OSM, ГИС ЖКХ, нормативы МНГП) в единую модель и довели её до интерактивной аналитической карты, а не статичных слайдов
- **Интерактивный инструмент моделирования** — пользователь может «поставить» новый парк/сквер прямо на карту и тут же увидеть, сколько людей он накроет; буфер строится корректно с учётом сферической геометрии, подсчёт населения мгновенный
- **Оптимизация** без фреймворков — карта быстро грузится, легко деплоится на GitHub Pages, понятна для дальнейшей доработки
- **Внимание к мелочам интерфейса**: кастомная масштабная линейка с русскими подписями, согласованные сдвиги зум-контролов и линейки при открытии правой панели, единая визуальная система цветов и шрифтов

## Что было самым сложным

- **Согласование данных из разных источников** — ГИС ЖКХ, OSM и нормативы МНГП используют разные классификации, и привязать категории ПРК к нормативам пешеходной доступности потребовало ручной работы
- **Выявление багов в JS** поскольку данный язык программирования для нас непривычен, процесс разработки интерфейса и его логики были не самыми простыми

## Что не получилось / Версия 2

Изначально планировалось не только показать **где** не хватает объектов ПРК, но и предложить **где их добавить**: выгрузить полигоны земельных участков и проанализировать свободные ЗУ как кандидатов под перспективные зелёные зоны для покрытия дефицитных территорий. От этой части пришлось отказаться — выгрузка и атрибутивная очистка ЗУ для всей Казани оказались слишком трудоёмкими в рамках сроков проекта.

**Идеи для версии 2:**
- Слой свободных земельных участков с автоматическим скорингом по дефициту обеспеченности
- Расширение инструмента «Добавление объекта»: после клика — **пересчёт показателей обеспеченности и процента покрытого населения на лету**, с обновлением карточек статистики в правой панели в реальном времени
- Возможность размещать несколько гипотетических объектов и сравнивать сценарии
- Учёт типа дорожной инфраструктуры (барьеры — реки, ж/д, автомагистрали) в построении буферов
- Уточнение модели населения за счёт данных переписи / операторов связи

## Использование нейросетей

Нейросети активно использовались **для генерации и отладки кода** веб-визуализации:
- Архитектура верстки и стилей правой/левой панелей, выезжающих по клику
- Реализация инструмента моделирования объекта (буфер на сфере, суммирование населения внутри)
- Геокодер на Nominatim, синхронизация контролов карты с состоянием панелей
- Отладка ошибок (конфликт `const`-объявлений, z-index слоёв, поведение при ресайзе)

Для **концептуального проектирования** (выбор показателей, методология обеспеченности, дизайн интерфейса) нейросети не использовались — это работа команды. Пространственный анализ в QGIS и расчёты в Python также сделаны вручную, дизайн реализован в Figma и экспортирован через Dev Mode

## Адаптивность

Интерфейс рассчитан на десктоп. Для экранов уже 1200 px панели расширяются с 25 % до 30 % ширины окна, для экранов уже 800 px — до 75 %, а шапка переносится в две строки.
